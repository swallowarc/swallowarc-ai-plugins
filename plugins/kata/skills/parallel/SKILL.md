# parallel（独立タスクの並列エージェント実行）

## 原則

独立した問題領域ごとに1エージェントを割り当てる。エージェントに自分のセッション履歴をそのまま継承させない。会話の流れを共有していれば楽ではあるが、実装者側の思い込みまで一緒に引き継いでしまう。渡すのは、そのタスクの判断に必要な情報だけを選んで構成したプロンプトである。これにより各エージェントは狭い範囲に集中でき、自分自身のコンテキストも並列化の調整作業のために温存できる。

## 使う条件／使わない条件

判断基準は1つ。**独立しているか、共有状態がないか。**

**使う:**
- 片方を修正しても、もう片方の結果に影響しない
- 各タスクが他のタスクの文脈を知らなくても着手できる
- 同じファイル・同じリソースを取り合わない

**使わない:**
- 失敗同士が関連していて、1つの修正が他にも波及しうる
- 着手する前にシステム全体の状態を把握する必要がある
- まだ何が壊れているか分かっていない探索段階

迷ったら「先に着手したタスクを完全に忘れても、残りのタスクを独立に進められるか」を自問する。答えが No なら並列化せず、まず1つのエージェント（または自分）で全体像を掴んでから分割する。

## 手順

### 1. 独立ドメインへの分割

対象を「互いに影響しない単位」に切り分ける。単位はテストファイルでもモジュールでも調査対象のバグ報告でもよいが、境界は「片方をいじってももう片方の正しさが変わらない」ことで引く。

### 2. 各エージェントへのプロンプト作成

各タスクについて、範囲・ゴール・制約・期待する出力形式を明記したプロンプトを個別に用意する。次の「良いプロンプトの条件」に従う。

### 3. 同一メッセージ内で並列起動

複数の Agent ツール呼び出しを**同一メッセージの中で**まとめて発行する。1メッセージにつき1呼び出ししか出さないと、そのぶんだけ逐次実行になり並列化の効果が消える。

### 4. 結果の統合

エージェントが返ってきたら、各サマリーを読み、変更同士が競合していないかを確認したうえで、全体テスト（または該当する検証コマンド）を実行してから統合する。この最終確認は省略できない（詳細は後述「統合時の検証」）。

## Claude Code 特化: isolation

ファイル変更を伴う並列タスクは、Agent ツールの `isolation: "worktree"` を指定して作業領域を分離する。分離しないまま複数エージェントに同じ作業ツリーで書き込ませると、片方の未コミットの変更をもう片方が壊す事故が起きる。読み取り専用の調査（原因究明・コードリーディングなど、ファイルを書き換えない作業）であれば `isolation` は不要である。

## 良いプロンプトの条件

焦点が1つに絞られていること・自己完結していること（エラーメッセージやファイルパスをそのまま貼る。「さっき話した件」のような参照に頼らない）・期待する出力形式を指定していることの3点を満たす。

**Bad:** 「コンポーネントの警告を全部直して」
**Good:** 「`src/components/Header.tsx` の ESLint 警告2件（62行目 `react-hooks/exhaustive-deps`、80行目 `no-unused-vars: theme`）を修正して。他のコンポーネントファイルは変更しないこと。修正箇所を diff 形式で返して。」

**Bad:** 「API のエラーを調べて直しておいて」
**Good:** 「`GET /api/orders/:id` が 500 を返す不具合を調査して修正して。再現条件と直近のエラーログは `logs/api.log` の 14:32 以降を参照。原因調査と修正内容の要約を返して。他のエンドポイントには触れないこと。」

**Bad:** 「ドキュメントをきれいに整理して」
**Good:** 「`docs/setup.md` のインストール手順セクションのみを、手順1〜5の番号付きリストに書き直して。`docs/faq.md` を含む他ファイルには触れないこと。変更後のセクションを差分で返して。」

## 統合時の検証

エージェントの「完了しました」「成功しました」という報告を、それ自体で証拠として扱わない。`kata:verify-done` に従い、diff を自分の目で確認し、全体テストを実行した結果をもって統合の可否を判断する。

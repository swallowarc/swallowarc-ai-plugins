# using-kata（スキル要否チェックの入口）

## 概要

応答や作業は、まず「使えるスキルはあるか」の判定を経てから動き出す。この判定を飛ばした瞬間、行動は場当たり的な思いつきに戻る。using-kata は、その判定を毎回強制するための入口である。

## サブエージェント除外

特定タスクの実行だけを目的に dispatch されたサブエージェントは、このスキルの対象外である。呼び出し元がすでにタスクとスキルの対応づけを終えた状態で仕事を渡しているため、サブエージェント側でスキル選定をやり直す必要はない。

## ルール: 該当する確率が 1% でもあれば起動する

応答・行動の前に、必ず「使えるスキルがあるか」を確認する。ここでいう行動には、明確化のための質問やコードベースの下調べも含まれる。判定の結果そのスキルが状況に合わなければ使わなくてよいが、「確認したかどうか」自体は省略できない。確認を飛ばして直接応答や作業に進むことは常に禁止と考える。

起動したスキルにチェックリストがある場合も対象である。TaskCreate/TaskUpdate が使える環境ではそれでタスク化し、使えない環境では todo リストで代替する。

## スキル早見表

| スキル | 起動タイミング |
|---|---|
| `kata:design` | 何か作る前 |
| `kata:plan` | 承認済み設計から |
| `kata:execute` | プランを実行 |
| `kata:tdd` | 実装コードを書く前 |
| `kata:debug` | バグ・失敗・予期しない挙動 |
| `kata:verify-done` | 完了を主張する前 |
| `kata:request-review` | タスク・機能完了時 |
| `kata:finish` | 実装完了後の統合判断 |
| `kata:parallel` | プランなしの独立タスク 2 つ以上 |

## 優先順位: プロセス系が先、実装系はその後

複数のスキルが同時に該当するときは、進め方そのものを決めるプロセス系スキル（`kata:design`・`kata:debug`）を先に起動し、その判断を受けてから実装系スキルに進む。

- 「作って」「実装して」と言われたら → まず `kata:design`
- 「直して」「動かない」と言われたら → まず `kata:debug`
- EnterPlanMode に入る前にも、先に `kata:design` を済ませておく

## Red Flags: これらの言い訳が浮かんだら止まる

| 心の声 | 実際のところ |
|---|---|
| ただの質問だから確認は要らない | 質問も行動の一種であり、判定の対象から外れない |
| 先にコードを見てから決めよう | 探索の進め方自体をスキルが規定している。見る前に確認する |
| この程度の作業にスキルは大げさ | 小さく始めた作業ほど後から肥大化しやすい。今のうちに確認する |
| このスキルは覚えているから読まなくていい | 記憶は最新版である保証がない。都度、現物を読む |
| 手を止めるのがもったいない | 判定にかかる時間より、判定を飛ばして手戻りする時間の方が長い |
| 今回だけは例外にしよう | 一度作った例外は、次回以降の言い訳の前例になる |

上のいずれかの思考が浮かんだら、それは判定を省略してよい合図ではなく、判定を今すぐ行えという合図である。

## ユーザー指示の優先

CLAUDE.md やユーザーからの直接指示は、スキルより優先する。ただし優先されるのは指示の内容であり、「スキルを確認する行為」そのものを省略してよいのは、ユーザーが明示的にスキル省略を求めたときに限る。

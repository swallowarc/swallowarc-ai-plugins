# request-review（レビュアーサブエージェントに検証を依頼する）

## 概要

自分の書いたコードを自分で読み返すだけでは、実装中の思い込みがそのまま見落としになりやすい。レビュアーサブエージェントには会話の履歴をそのまま渡さず、判定に必要な情報だけを選んで渡す。会話の流れを共有していないぶん、実装者側の事情に引きずられない評価が返ってくる。

レビューは完成後に一度だけまとめて行うものではない。差分が小さいうちに刻んで依頼したほうが、問題が見つかったときに直す範囲も検証し直す範囲も小さく済む。

## 役割分担（Claude Code 特化）

バグや実装品質の粗探しは、ビルトインの `/code-review` が利用可能な環境ではそちらに委譲する。本スキルが主眼を置くのは **spec・プランへの適合性**、つまり「要求されたことを満たしているか」と「要求されていないものまで作り込んでいないか」の両面である。過不足のどちらも見逃さない。

ビルトインが利用できない環境では、`references/code-reviewer-prompt.md` のテンプレートで spec 適合とコード品質の両方をまとめてカバーする。

## 必須タイミング

- `kata:execute` の各タスク完了後
- 大きな機能がひとまとまり完了した後
- main へのマージ前

## 手順

### 0. ビルトイン `/code-review` が利用可能なら先に実行する

利用可能な環境では、まず `/code-review` を実行してバグ・実装品質の指摘を得る。この場合、以降の手順は spec・プラン適合の確認に専念する。利用できない環境ではこのステップを飛ばし、以降の手順のテンプレートで品質面も含めてカバーする。

### 1. BASE_SHA・HEAD_SHA を確定する

```bash
HEAD_SHA=$(git rev-parse HEAD)
```

`BASE_SHA` を `git rev-parse HEAD~1` で機械的に求めない。1つのタスクが複数コミットにまたがる場合、`HEAD~1` は直前の1コミットしか指さず、タスク全体の差分を取りこぼす。該当タスクに着手する前に控えておいた（進捗の記録やメモに残しておいた）コミット SHA を `BASE_SHA` として使う。

### 2. 差分をファイルに書き出す

```bash
mkdir -p .kata
git diff -U10 "$BASE_SHA".."$HEAD_SHA" > ".kata/review-${BASE_SHA}-${HEAD_SHA}.diff"
```

`.kata/` 配下に一意な名前（例: `.kata/review-<BASE>-<HEAD>.diff`）で書き出す。`kata:execute` のタスクレビューと同じ流儀であり、同時に複数のレビュー差分を扱っても上書きし合わない。会話履歴ではなくこのファイルをレビュアーに読ませる。

### 3. レビュアーサブエージェントを dispatch する

Agent ツール（`general-purpose`）で `references/code-reviewer-prompt.md` の各プレースホルダを埋め、そのままレビューを依頼する。テンプレートは「何を実装したか」「何を満たすべきか」「どの diff を見るか」をレビュアーに過不足なく渡すために存在する。プレースホルダが残ったまま dispatch しない。

### 4. フィードバックに対応する

- **Critical**: その場で即修正する。次の作業へ進まない
- **Important**: 先に進む前に修正する
- **Minor**: 記録だけして先へ進んでよい
- レビュアーの指摘が誤っていると判断した場合、黙って従うのでも無視するのでもなく、コードやテストの実際の挙動を根拠に反論する

## Red Flags

- 「この変更は単純だからレビュー不要」と判断してスキップする
- Critical な指摘を残したまま次の作業へ進む
- レビュアーの指摘に根拠なく反論する、または根拠を確かめず盲従する

## 関連スキル

- `kata:execute` の各タスク完了後・ブランチ全体の最終レビューから呼び出される
- ビルトイン `/code-review` が使えない環境の代替手段として `references/code-reviewer-prompt.md` を使う

# quick（小型タスクのライトレーン）

## 概要

quick は、design→plan→execute の工程構造を通すほどではない小型タスクを、kata 最小限の規律で完結させるスキルである。「作る → `kata:design` 起点」「直す → `kata:debug` 起点」に並ぶ第 3 の入口、「小型 → quick 起点」を担う。

軽くなるのは工程であって規律ではない。設計対話・プラン文書・worktree 分離・サブエージェントレビューは省くが、実装は `kata:tdd`、完了主張は `kata:verify-done` のゲートをそのまま通す。省いてよいのは往復であって、検証と証拠ではない。

## 鉄の掟

```
発動はユーザーの明示呼び出しのみ。
小型でないと判明したら、直ちに手を止めて design / debug へ切り替える。
```

サイズの分類は、ユーザーが呼び出しの時点で行っている。モデルの側でタスクを「小型だから quick で十分」と自己分類して起用することは、using-kata の除外規定とこの掟の両方への違反である。quick の役割は分類のやり直しではなく、実行中の逸脱検出である。

## タスクループ

1 タスクにつき次の 8 手順を踏む。複数タスクを渡されたら、1 つずつこのループを完結させて直列で回す。

1. **受付と分割** — 依頼を「独立に検証・コミットできる単位」に分割し、TaskCreate（利用できなければ todo リスト）に登録する。2 単位以上あるときは「直列で処理する」と 1 行宣言する（返答は待たない）。独立タスクを並列で流したい場合の入口は `kata:parallel` であると 1 行案内する。
2. **意図確認（最大 1 問）** — 曖昧さが 1 問で解けるなら、AskUserQuestion（利用できなければ本文の番号選択）で聞いてから着手する。1 問で解けない曖昧さは、小型ではない兆候としてエスカレーションする。
3. **サイズ自己判定** — 着手前に「設計判断を伴わないか」「変更見込みがバックストップの閾値内か」を確認する。超えそうなら、この時点でエスカレーションする。
4. **実装** — コードを触るタスクは、Skill ツールで `kata:tdd` を起動して従う。記憶にある内容で済ませない。設定ファイルなどは tdd 自身の例外規定（ユーザー確認）で扱う。typo・ドキュメントなどテスト対象が存在しない変更は、tdd の適用外として直接編集する。
5. **検証** — Skill ツールで `kata:verify-done` を起動して従う。テストがない変更でも、実挙動・実表示の確認（ビルド実行、該当コマンドの実行結果）で新鮮な証拠を取る。
6. **セルフレビュー** — コミット前に diff を読み直し、変更が意図どおりか、無関係な変更が混ざっていないかを確認する。
7. **コミット** — 今回のタスクで触ったファイルだけを明示的に `git add` して、1 タスク 1 コミットにする。push はしない。quick はブランチを切らず現在のブランチで作業するため、この限定 add が、作業ツリーに元からあった無関係な変更を巻き込まないための唯一の防波堤である。`git add -A` や `git add .` は使わない。
8. **報告** — タスクの結果と検証証拠を報告して終了する。`kata:finish` は呼ばない。

## エスカレーション

### 質的トリガー（即・強制切替）

次のいずれかに当たったら、その場で編集を止める。

- 設計判断が必要になった（複数の実装方針から選ぶ必要が出た、インターフェースの新設・変更が必要になった）→ `kata:design`
- 想定外のバグに遭遇した（依頼と別の不具合を見つけた、テストが想定外の理由で落ちた）→ `kata:debug`
- 意図確認が 1 問で解けなかった → `kata:design`

### 数値バックストップ（停止して確認）

「これは設計判断ではない」という合理化を機械的に補足するガードである。

- 1 タスクの変更が 3 ファイルを超えたら（テストファイル含む）停止する
- 同一タスクで「修正 → 検証 → 失敗」のサイクルが 2 回続いたら停止する（検証は tdd の GREEN 検証・verify-done のどちらも数える。3 回目の修正試行に入らない）

バックストップには誤爆がありうる（正当な 4 ファイル変更など）。質的トリガーと違い即切替はせず、停止してユーザーに続行可否を確認する。続行が選ばれたら、そのタスクに限りバックストップを解除して続ける。

### 切替手順（質的トリガー時）

1. それ以上の編集をしない。
2. 状況を報告する——何のタスクで、どこまで進み、何が起きたか。ここまでの変更ファイルと diff の要旨。
3. AskUserQuestion（利用できなければ本文の番号選択）で「変更を保持したまま切替」「変更を破棄して切替」の 2 択を提示し、選択を受けてから `kata:design` / `kata:debug` を起動する。
4. コミット済みの完了タスクは確定として残す。エスカレーションが作用するのは現在のタスクだけであり、未着手の残りタスクは design / debug の完了後にユーザーが改めて判断する。

## Red Flags

- 「もう少しで終わるから」と閾値超過後も編集を続けている
- 設計判断を「実装の詳細」と言い換えて quick に留まろうとしている
- 見つけたバグを debug を経ずに「ついでに」直そうとしている
- 検証失敗 2 回のあと「もう 1 回だけ」と考えている

tdd・verify-done の Red Flags は、各スキルを起動したときにそちらが持ち込む。本スキルには重複させない。

## 関連スキル

- 実装のゲートは `kata:tdd`、完了主張のゲートは `kata:verify-done` を都度起動して従う。
- エスカレーション先は `kata:design`（設計判断）と `kata:debug`（想定外のバグ）である。
- プランなしの独立タスクを並列で流す入口は `kata:parallel` である。quick からは案内のみ行い、自動では跨がない。

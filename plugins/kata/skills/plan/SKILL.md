# plan（実装計画を最小タスクへ分解する）

## 概要

plan は、承認済みの設計や要求定義を、実装者がそのまま手を動かせる粒度のタスク列に変換するスキルである。想定読者は「このコードベースの文脈を一切持たない熟練エンジニア」であり、腕はあっても、このプロジェクトの流儀もクセも知らない。触るファイル・書くコード・書くテスト・実行して確認するコマンドまで、プランの中に全部書く。読み手に推測させてはならない。

プラン全体を貫く原則は次の4つである。

- **DRY** — 同じ説明・同じコードを複数タスクに重複させない
- **YAGNI** — 今回のスコープに要らない機能をタスクに混ぜない
- **TDD** — コードを書くタスクは必ず「失敗するテスト→実装→成功確認」の順で組む
- **高頻度コミット** — タスクの内部でも、意味のある単位ごとにコミットを刻む

## 保存先

プランは `docs/kata/plans/YYYY-MM-DD-<feature-name>.md` に保存する。ユーザーが別の保存先を指定した場合はそちらを優先する。

## spec 受け入れ検査

plan 着手の最初のアクションとして、入力の spec を検査する。

1. **証跡の確認** — spec に「検討済み事項」節があるか。なければ未確定点の棚卸しの証跡が確認できない spec である。拒否はせず、「このまま plan に進めるか、`kata:design` で棚卸しを行うか」をユーザーに確認する
2. **内容の検査** — spec を通読し、設計判断を左右する未確定・複数に読める記述・矛盾がないかを確認する。観点は棚卸しと同じレンズ（境界値・失敗時の挙動・同時実行・互換性・性能・外部依存の実挙動）を使う。検討済み事項に載っている項目は未確定として問い直さないが、spec 本文との矛盾を見る検査からは除外しない

見つけた未確定を自分の推測で埋めてタスク化することは、プレースホルダと並ぶプランの欠陥である。見つけた項目は着手前に一度だけまとめてユーザーに問い（タスク作成中に気づくたびに割り込まない）、回答が設計の根幹を変える場合は plan を続けず `kata:design` への差し戻しを提案する。根幹を変えない回答は、plan を続ける前に spec の「検討済み事項」（必要なら本文も）へ反映し、反映箇所が spec の他の記述と矛盾しないことを確認する。spec に反映されていない決定を plan だけが持つ状態を作らない。

## スコープチェック

着手前に、対象の spec が複数の独立したサブシステムにまたがっていないかを確認する。またがっている場合、1 本のプランに詰め込まず、サブシステムごとにプランを分割する。分割の基準は「そのプラン単独で動作確認とテストが完結するか」であり、完結しないなら境界の引き方が誤っている。

## ファイル構成の先行設計

タスクを定義する前に、今回の実装で新規作成・変更するファイルを洗い出し、各ファイルの責務を一覧化する。

- 1 ファイルには 1 つの責務だけを持たせる。複数の役割を担わせない
- 既存コードに手を入れる場合、まずそのプロジェクトの流儀を確認し、無関係な範囲までついでに整理しない
- この一覧がそのままタスク分割の下敷きになる。ファイル一覧を作らずにタスクへ進まない

## タスクの適正サイズ

タスクは「独自のテストサイクルを持つ最小単位」である。境界を引く基準は、レビュアーが隣のタスクを承認しながら、このタスクだけを差し戻せるかどうかにある。差し戻せないほど密結合な変更は 1 タスクにまとめ、逆に無関係な変更を無理に同居させない。

セットアップ・設定・雛形作成のような小さな下準備は、それを必要とする本体タスクに含めてよい。わざわざ単独タスクに切り出す必要はない。

## ステップ粒度

1 ステップは 1 アクションであり、所要時間の目安は 2〜5 分である。粒度が大きすぎるステップは分割する。典型的な内訳は次の通り。

1. 失敗するテストを書く
2. 実行して失敗を確認する
3. 最小限の実装を書く
4. 実行して成功を確認する
5. コミットする

この 5 つを 1 セットとして、タスク内の挙動ごとに繰り返す。

## プランヘッダのテンプレート

プラン文書の冒頭は、必ず次の形式で始める。

```markdown
# <機能名> 実装計画

> **For agentic workers:** 本プランの実行には `kata:execute` が必須のサブスキルである。タスクは `- [ ]` のチェックボックスで進捗を追跡する。

**Goal:** <この実装が何を達成するかを1文で>

**Architecture:** <採用するアプローチを2〜3文で>

**Tech Stack:** <使用する主要な技術・ライブラリ>

## Global Constraints

<spec が定めるプロジェクト全体の約束事をここへ書き写す。持ち込み禁止のライブラリ、
公開 API の互換性維持、ユーザー向け文言の表記規則、動作保証するランタイムの版、
といった類のもの。数値や名前は要約せず spec の記載どおりに写す。
各タスクの要件には、明記がなくてもこの節の内容が常に含まれる。>

---
```

## タスク構造のテンプレート

各タスクは次の形式で書く。

````markdown
### Task N: <コンポーネント名>

**Files:**
- Create: `src/shipping/fee.ts`
- Modify: `src/shipping/index.ts:12-18`
- Test: `tests/shipping/fee.test.ts`

**Interfaces:**
- Consumes: <前段のタスクが Produces で公開した関数・型を、名前と型を変えずにそのまま使う>
- Produces: <後続タスクが依存する、このタスクが公開する関数・型。他のタスクの担当者はこのタスク本文まで読まないことが多く、ここに書いた名前と型がそのまま外部仕様になる>

- [ ] **Step 1: 失敗するテストを書く**

```typescript
test("重量が5kg以下なら送料500円を返す", () => {
  expect(calculateShippingFee(5)).toBe(500);
});
```

- [ ] **Step 2: 実行して失敗を確認する**

Run: `npx vitest run tests/shipping/fee.test.ts`
Expected: FAIL — `calculateShippingFee is not defined`

- [ ] **Step 3: 最小実装を書く**

```typescript
export function calculateShippingFee(weightKg: number): number {
  return 500;
}
```

- [ ] **Step 4: 実行して成功を確認する**

Run: `npx vitest run tests/shipping/fee.test.ts`
Expected: PASS

- [ ] **Step 5: コミットする**

```bash
git add tests/shipping/fee.test.ts src/shipping/fee.ts
git commit -m "feat: 5kg以下の送料計算を追加"
```
````

## プレースホルダ禁止

次のような表現がプランに残っていたら、それはプランの失敗である。見つけた時点で書き直す。

- 「Task N と同様」という省略。実装者はタスクを番号順に読むとは限らないので、必要なコードは各タスクに完全な形で再掲する
- 「適切にエラー処理」のように、何をどう処理するのかを読み手に決めさせる指示
- 「TBD」「後で実装」をはじめとする先送り表現。プランは保存した時点で完結していなければならない
- コードを書くステップにコードブロックが添えられていない状態。散文でやり方を語っても、書くべきコードの代わりにはならない。「テストを書く」というステップなら、テストコードそのものを載せる
- プランのどこにも定義が現れない型・関数・変数を、Interfaces やステップ中のコードが参照している状態

## セルフレビュー

プランを書き終えたら、spec を読み直してプランと突き合わせる。サブエージェントに投げる作業ではなく、自分の目で行う。

1. **spec カバレッジ** — spec の各要件に対応するタスクを 1 つずつ指させるか確認する。抜けがあればタスクを追加する
2. **プレースホルダ走査** — 「プレースホルダ禁止」節に挙げたパターンが残っていないか検索する
3. **型・シグネチャの一貫性** — 後続タスクの Interfaces で使っている関数名・型名・引数が、それを Produces したタスクの定義と一致しているか確認する。あるタスクでは戻り値を `ShippingResult` 型で定義したのに、別のタスクでは `{ fee: number }` という素のオブジェクトとして扱っている、といった食い違いは見落としやすい不整合の典型である

見つけた問題はその場で inline 修正する。往復レビューにはしない。

## 実行への引き継ぎ

プランを保存したら、次の形で `kata:execute` の起動を提案する。

> 「plan を `docs/kata/plans/<filename>.md` に保存しました。`kata:execute` を起動して実装を進めますか？実行モード（自走 / サブエージェント駆動）は execute が判断して宣言しますが、希望があればここで指定してください。」

実行方式をここで質問はしない。判断は `kata:execute` 側の役割である。ただしユーザーがこの時点で方式を指定した場合は、その指定が execute での判断に優先する。

## 関連スキル

- 承認済み spec の作成は `kata:design` が担当する。plan はその成果物を入力として受け取る
- 保存したプランの実行は `kata:execute` に引き継ぐ

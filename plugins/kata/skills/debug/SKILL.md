# デバッグ（体系的根本原因調査）

## 概要

**中核原則:** 修正の前に必ず根本原因を特定する。症状への対処は失敗である。

原因を特定しないまま当てずっぽうで直したコードは、たまたま今回のケースを黙らせるだけで、同じ原因から生まれる次の不具合を止められない。むしろ触った箇所が増える分、新しい不具合を生む確率のほうが上がる。「急いでいるから」「簡単そうだから」は原因調査を省略してよい理由にはならない。急ぐほど、当てずっぽうの修正がやり直しを増やして遅くする。

このスキルが定めているのは手順の文言ではなく、「理解していないものを直さない」という姿勢そのものである。手順だけなぞって「調査した体で修正案を出す」のは、手順違反である以上に姿勢の違反として扱う。

## 鉄の掟

```
根本原因の調査なしに修正しない
```

Phase 1（下記）を完了するまで、修正案を1つも提案してはならない。「原因はまだ分からないが、たぶんこれで直る」という提案自体が、この掟への違反である。気づいた時点で提案を取り下げ、Phase 1 に戻る。

## 適用場面

エラー、テスト失敗、想定外の挙動、性能劣化、ビルド失敗、結合不良——技術的な問題全般に適用する。

特に次のときこそ使う。省略したくなる場面ほど、省略した代償が大きい。

- 時間的プレッシャーがあるとき（焦りは当てずっぽうを誘う）
- すでに複数の修正を試したとき（それでも直っていないなら、理解が足りていない証拠）
- 原因が簡単に見えるとき（簡単に見える不具合にも根本原因はある）
- 「前も似た不具合があって、そのときはこう直した」と記憶で判断したくなるとき（前回と今回が同じ原因である保証はない）

いずれも「今回は例外として省略してよい」と感じやすい場面である。その感覚自体が調査を省略する合理化であり、正当な例外ではない。

## 4つのフェーズ

各フェーズを完了するまで次のフェーズへ進まない。

### Phase 1: 根本原因調査

修正案を出す前に、以下を順に行う。

1. **エラーメッセージを最後まで読む** — スタックトレース、行番号、エラーコードを読み飛ばさない。答えがそこに書かれていることは多い。
2. **確実に再現させる** — 毎回同じ手順で発生するか確認する。再現しないなら、まだ「直す」段階ではなく「情報を集める」段階である。
3. **直近の変更を確認する** — `git diff`、直近のコミット、新しく追加した依存、設定変更を洗い出す。壊れる直前に何が変わったかは最有力の手がかりになる。
4. **多層システムでは層の境界で証拠を集める** — 処理がAPI・キュー・DBのように複数コンポーネントをまたぐ場合、いきなり怪しい層を推測しない。各境界にログを仕込んで一度実行し、「どの層まではデータが正しく届いていて、どの層から壊れているか」を証拠として先に確定させる。推測はそのあとでよい。

   例えば CSV 一括取り込みバッチが「アップロードはできるのに DB に反映されない」場合:

   ```bash
   # Layer 1: アップロード受付（Nginx）
   echo "=== 直近のアップロードリクエスト ==="
   grep "POST /api/imports" /var/log/nginx/access.log | tail -5

   # Layer 2: パーサープロセス
   echo "=== パーサーが受け取ったペイロード ==="
   grep "job_id=$JOB_ID" /var/log/importer/parser.log | tail -20

   # Layer 3: キュー投入
   echo "=== キューへの投入内容 ==="
   redis-cli LRANGE "queue:import:$JOB_ID" 0 -1

   # Layer 4: DB反映
   echo "=== 該当ジョブの書き込み件数 ==="
   psql -d app_production -c "select count(*) from imported_rows where job_id = '$JOB_ID';"
   ```

   これを1回実行するだけで、「Layer 1・2 は正常、Layer 3 のキューが空」のように壊れている層が一目で分かる。層を跨いで推測するより、この4行のほうが速い。
5. **発生源へ遡る** — 壊れている層が分かったら、その層で症状（symptom）を直すのではなく、悪い値がどこで生まれたかを呼び出し元へ遡って特定する（詳細な手順は後述の「逆方向トレース」）。修正は発生源（source）に対して行う。

### Phase 2: パターン分析

1. **動く類似例を探す** — 同じコードベースの中に、壊れている処理と似た役割で正常に動いているコードがないか探す。
2. **参照実装は全部読む** — 既存パターンに乗る場合、そのパターンを流し読みで済ませない。全行読んで、なぜそう書かれているかまで理解する。「長いから該当箇所だけ拾う」は理解不足のまま適用する結果を招く。
3. **差分を全部列挙する** — 動くコードと壊れているコードの違いを、些細に見えるものも含めてすべて書き出す。「そこは関係ないはず」という判断を先に下さない。設定値1つ、実行順序1つの違いが原因であることは珍しくない。

### Phase 3: 仮説と検証

1. **仮説を1つずつ言葉にする** — 「Xが原因だと考える。理由はY」の形で明文化する。頭の中だけで済ませない。例:「デプロイ直後だけキャッシュを外しても同じ不具合が消えないため、キャッシュではなく設定読み込みの順序が原因だと考える」。
2. **最小の変更で検証する** — 仮説を確かめるのに必要な、いちばん小さい変更だけを加える。一度に変える変数は1つ。検証用の変更と、実際の修正を兼ねようとしない。
3. **ダメなら新しい仮説を立てる** — 検証が外れたら、その変更の上にさらに手を加えて帳尻を合わせようとしない。変更を戻し、新しい仮説を1から立てる。
4. **分からないときは「分からない」と言う** — 分かったふりをして次に進まない。分からないまま進めるくらいなら、その旨を明言して調査を続けるか相談する。

### Phase 4: 実装

1. **失敗する再現テストを先に作る** — `kata:tdd` に従い、今回のバグを検出できる最小のテストを書き、それが失敗することを確認する。
2. **根本原因への修正を1つだけ行う** — Phase 1〜3 で特定した原因だけを直す。「ついでに気になった箇所」の改善や同時リファクタリングはここでは行わない。
3. **検証する** — `kata:verify-done` に従い、作成したテストが通り、既存のテストも壊れていないことを確認してから完了を主張する。

### 4フェーズの要約

| フェーズ | やること | 次に進んでよい条件 |
|---|---|---|
| Phase 1: 根本原因調査 | エラーを読む・再現させる・変更履歴を確認・層ごとに証拠を集める・発生源へ遡る | 「何が」「なぜ」壊れているかを説明できる |
| Phase 2: パターン分析 | 動く例を探す・参照実装を全読・差分を列挙 | 動く/壊れるの違いを具体的に言える |
| Phase 3: 仮説と検証 | 仮説を明文化・最小変更で検証 | 仮説が確認された、または反証された |
| Phase 4: 実装 | 失敗するテストを作る・原因を1つだけ直す・検証する | テストが通り、既存テストも壊れていない |

## 3回失敗したら

同じ不具合への修正を3回試して3回とも失敗したら、Phase 1 に戻って再調査するのではなく、いったん立ち止まってアーキテクチャそのものを疑う。

「直すたびに、直した場所とは別の場所に新しい症状が出る」という繰り返しは、個々のバグの問題ではなくアーキテクチャの問題である兆候である。この場合は4回目の修正を試みる前に、状況を整理してユーザーと議論する。

## Red Flags

次のように考え始めたら、その場で手を止めて Phase 1 に戻る。理由を検討する必要すらない。

- 「とりあえず変えてみる」
- 「複数の変更をまとめて試してから様子を見る」
- 「たぶんXが原因だから、それを直す」
- 調査より先に修正案を列挙し始めている
- 「よく分からないが、これで動く気がする」
- 「参考にしたパターンとは違うやり方だが、たぶん大丈夫」
- テストを書かずに「手で動かして確認」しようとしている
- すでに2回失敗しているのに「もう1回だけ」と考えている
- 「原因は分からないが、とりあえずログを増やして様子を見る」で調査を終えたつもりになっている
- 差分を1つ見つけた時点で、残りを確認せずに「これが原因」と決めつけている

これらはすべて、調査を飛ばして修正に着地しようとする動きである。1つでも当てはまったら、その行為を止めて全て Phase 1 へ戻る。

## 合理化への反論

| 言い分 | 反論 |
|---|---|
| 単純な問題だから調査は要らない | 単純に見える不具合にも必ず原因がある。調査自体は数分で終わる規模のことが多い。 |
| 緊急だから今すぐ直したい | 当てずっぽうの修正は外れるたびにやり直しが発生する。急ぐ状況ほど一発で当てる必要があり、それには調査が要る。 |
| まず試してから、必要なら調査する | 最初の一手がその後の方針を決めてしまう。調べずに打った一手は外れても学びが少ない。 |
| テストは修正が効いたのを確認してから書く | 順序が逆だと、テストは「今のコードがそう動く」ことしか確認できず、直したい不具合を検出できる保証がない。 |
| 複数箇所を同時に直したほうが速い | どの変更が効いたのか切り分けられなくなる。原因不明のまま「なぜか直った」状態は再発する。 |
| もう1回だけ修正すれば直りそう | 2回外した仮説の延長線上での3回目は、たいてい同じ理解不足の繰り返しである。3回目に進む前に「3回失敗したら」の節に従う。 |
| ログを足しただけで満足していい | ログは証拠を集める手段であり、証拠を読んで原因を特定するところまでやって初めて Phase 1 が終わる。 |
| このパターンは前にも見たので調査は省略できる | 症状が似ていても原因が同じとは限らない。今回のコードで実際に確認するまでは推測でしかない。 |

## 原因不明のとき

Phase 1〜3 を実際に一通りやり切ったうえで、それでも環境依存やタイミング依存としか説明がつかないことはある。その場合は次を行う。

1. どのフェーズで何を調べ、何が分かって何が分からなかったのかを記録に残す。次に同じ現象に遭遇した人（未来の自分を含む）が調査をゼロからやり直さずに済むようにする。
2. 再発を止められないなら、リトライ・タイムアウト・エラーメッセージの改善・ログの強化といった実務的な緩和策を入れる。これは原因を諦めることではなく、原因不明のまま被害を抑える手当てである。

ただし「原因不明」で片付けられたケースの 95% は調査不足である。原因不明と結論する前に、4つのフェーズをすべて踏んだと胸を張って言えるか自問する。

## 補助テクニック

### 逆方向トレース

エラーが呼び出しの深いところで起きているとき、止まった場所をそのまま直すと症状潰しになる。

- まず、エラーを直接引き起こしたコードを特定する。
- 「これは誰に呼ばれたか」「呼び出し元は何を渡したか」を1段ずつ遡る。
- 最初に怪しい値が生まれた地点に行き着くまで繰り返す。途中で止めない。
- 手で追えないほど深い場合は、疑わしい処理の直前に呼び出し履歴つきのログを一時的に仕込み、1回実行して証拠を得てから追跡を再開する。
- 修正は必ずこの発生源に対して行う。エラーが「現れた場所」だけを直さない。

### 多層防御

発生源を直しても、別の呼び出し経路・モック・将来のリファクタリングが同じ不正な値をすり抜けさせる余地は残る。根本原因の修正後、その値が通過する各層に検証を足す。

- 受け口: API 境界で明らかに不正な入力を拒否する。
- 業務ロジック: その操作にとって意味をなさない値を弾く。
- 実行環境ガード: テスト環境など特定の文脈でだけ危険になる操作を拒否する。
- ログ計装: 他の層をすり抜けたときに原因追跡できる記録を残す。

目標は「バグを直した」ではなく「同じバグが構造的に起こり得ない」状態である。1箇所の検証で満足しない。

### 条件ベース待機

「Xミリ秒待てば終わっているはず」という決め打ちの `sleep` は、実行環境の速度差でテストや同期処理をフレーキーにする典型的な原因である。待ちたいのは時間ではなく状態である。

- 知りたい条件（値がセットされた・処理が完了した・ファイルができた等）を直接ポーリングする。
- タイムアウトを必ず設け、超過時は「何を待っていたか」が分かるエラーを出す。
- 条件の評価はループの中で毎回行う。ループ前に取った値を使い回さない。
- 任意の `sleep` を置いてよいのはタイミングそのものを検証する場合だけで、その際は根拠をコメントに残す。

## 関連スキル

- 再現テストの作成は `kata:tdd` の RED フェーズに従う。
- 修正後の完了確認は `kata:verify-done` に従う。

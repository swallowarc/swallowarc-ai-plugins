# 検証してから完了を主張する

## 鉄の掟

```
新鮮な検証の証拠なしに完了を主張しない
```

このメッセージの中で検証コマンドを実行していないなら、「通った」「直った」「動いた」とは言えない。過去の実行結果も「たぶん通る」という見込みも証拠にはならない。検証していないのに完了を主張するのは、効率を優先した判断ではなく、確かめていないことを確かめたと言っている虚偽である。

## ゲート手順

完了・成功・満足を口にする前に、必ず次の順で踏む。どれか一つでも飛ばしたら、それは検証ではなく嘘である。

1. **特定** — その主張を証明できるコマンドは何か。
2. **実行** — 省略や部分実行ではなく、そのコマンドを最後まで走らせる。
3. **精読** — 出力全体と exit code を読む。都合のいい行だけ拾わない。
4. **照合** — 出力は主張と一致しているか。一致していなければ実際の状態をそのまま報告する。
5. **主張** — ここまで終えて初めて、証拠付きで結果を述べてよい。

## 主張と必要な証拠の対応表

| 主張 | 必要な証拠 | 証拠にならないもの |
|---|---|---|
| テストが通った | テストコマンドの出力（failures 0） | 過去の実行結果、「通るはず」という見込み |
| ビルドが成功した | ビルドコマンドの exit code 0 | linter が通ったこと |
| バグを直した | 元の症状を同じ条件で再テストした結果 | コードを変更したという事実だけ |
| エージェントの作業が完了した | VCS の diff を自分で確認した内容 | エージェント自身の「成功しました」という報告 |
| 要件を満たした | プランと突き合わせたチェックリスト | テストが通っていることだけ |

## Claude Code 特化: 実動作の検証

コマンドの exit code だけでは、実際の画面や挙動まで確認できないことがある。ビルトインの `verify` スキルが利用可能なら、それを使って対象の機能を実際に動かし、結果を観察する。利用できない環境では、対象のフローを自分の手で実行し、出力や画面を目で見て確認してから完了を主張する。

## Red Flags

次の言い回しが浮かんだ、あるいは書きかけたら、その場で手を止めて検証コマンドを実行する。

- 「〜のはず」「たぶん通る」「問題ないと思う」
- 検証より先に「Done!」「完了しました」と書いている
- 一部だけ確認した結果を全体の結論として扱っている
- 検証せずにコミット・PR作成へ進もうとしている

## 合理化への反論

| 言い分 | 反論 |
|---|---|
| 動くはずだから大丈夫 | 「はず」は証拠ではない。実行して確かめる。 |
| 自信がある | 自信の強さと実際の正しさは無関係である。 |
| 今回だけは省略する | 一度認めた例外は次回以降の基準になる。 |
| linter は通った | linter はコンパイルもテストも保証しない。 |
| エージェントが成功と言った | 自己申告は第三者による確認ではない。diff を自分で見る。 |
| 疲れたのでここで終わりにしたい | 疲労は検証を省略してよい理由にならない。 |

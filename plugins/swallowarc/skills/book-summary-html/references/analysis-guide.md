# 分析ガイド（フェーズ1）

生成前に行う分析。結果をまとめてユーザに提示し、合意を得てから生成する。

## 1. 素材の把握

- 素材の場所・形式を確認する（章別 mindmap、章別 memo、本文抜粋、章立て一覧など）。
- どれを**主ソース**にするか決める。複数あるなら主従を決める（例：mindmap を主、memo を解説の補強に使う）。
- 素材が本のどの範囲を網羅しているか（全章あるか、抜けがあるか）を確認する。

## 2. ターゲット読者層の予想

本の内容・前提知識・語り口から、**想定読者を予想して明示する**。判断材料：

- 前提として要求される知識レベル（入門書か、実務者向けか、専門家向けか）
- 扱うテーマの抽象度（概念中心か、具体的手法中心か）
- 原書の対象読者の記述があればそれを尊重する

予想した読者像を**1〜2文で言語化**し、ユーザに提示する。要約の語彙・補足の量・解説の丁寧さはこの読者像に合わせる。

> 例：「DevOps を体系的に学び始めるエンジニア／チームリーダー。個々のツールの知識はあっても、原則と全体像はこれから整理する層」

## 3. 著者の主張の抽出

各部・各章について、**著者が主張している核**を抽出してリスト化する。

- 1章あたり **1〜3個** の箇条書きにする。
- 「何を述べているか（トピック）」ではなく「**何を主張しているか（クレーム）**」を書く。
  - 弱い例（トピック）：「バリューストリームについて説明している」
  - 強い例（クレーム）：「仕事は『顧客に価値が届くまでの流れ』として捉え直すべきで、改善はまず時間とムダの計測から始まる」
- このリストは要約で**主張を漏らさないためのチェック基準**になる。フェーズ2・3で参照する。

抽出した主張リストもユーザに提示し、認識が合っているか確認するとよい。

## 4. ファイル分割単位の提案

抽出した構造（部・章の数と関係）を踏まえ、分割単位を提案する。

- 候補：**章ごと** / **部ごと** / **その他**（テーマ単位など、本の構造上わかりやすい切り方があれば）
- それぞれの長所短所を簡潔に添え、**推奨を先頭**に置く。
- 章数・部数の規模感も判断材料にする（例：24章を章ごとにすると 24＋index ファイルで多すぎる→部ごとが妥当、など）。

## 5. index 用の2ブロックを用意する

フェーズ2の index には、全体像と各部カードの間に次の2ブロックを置く。

- **この本がターゲットとする読者**：2 で予想した読者像を、`.persona-grid` の数枚のカード（役割＋その読者が得るもの）として示す。本の内容からの推定であり、末尾に「（本書の内容から推定）」と添える（AIバッジは不要）。
- **おすすめの読み進め方**：AI が本の構成・主張を分析した独自の提案。立場別の読む順序などを示す。`.ai-analysis` ＋ AIバッジで本書外と明示する（`writing-rules.md` の例外条項に従う）。本の構成と整合させ、事実誤りを書かない。

## 提示のまとめ方

次を1つのメッセージにまとめて提示し、合意を求める：

1. 主ソースと対象範囲
2. 予想したターゲット読者像
3. 各章の著者の主張リスト（または代表例＋全体方針）
4. ファイル分割案（推奨＋代替）と、生成するファイル一覧

合意が取れるまで生成に進まない。

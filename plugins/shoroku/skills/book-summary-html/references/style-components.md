# スタイル部品の一覧と使いどころ

`assets/style.css` が提供するグラフィカル部品。視認性と理解しやすさのために使う。
**注意**：部品に入れる文言も `writing-rules.md` に従う（本にない情報・比喩・誇張を入れない）。

各ページは `<style>:root{ --accent:#色; --accent-soft:#薄い色; }</style>` でアクセント色を設定する（部ごと・章ごとに色分けすると見分けやすい）。

## 基本構造

- `.topbar` … 上部固定ナビ（目次＋他ページへのリンク）。リンク群は `.nav`（`.nav-checkbox`＋`.nav-toggle`＋`.nav-links`）で囲む。デスクトップは横並びで常時表示、スマホでは折りたたみ、`.nav-toggle`（「章 ▾」）のタップで全幅ドロップダウンとして開く（JS不要・章数が多くてもヘッダ高さは1行ぶんに固定）
- `.hero` … ページ見出し（`.eyebrow` ラベル＋`h1`＋`.lead` 説明）
- `.toc` … ページ内目次（章リスト）
- `.chapter` + `.chapter-head`（`.num`＋`h2`）… 章ブロック
- `.topic` … 「要点＋解説」の基本カード。`h3.ic`（`.emoji` 枠つき）＋`.summary`＋`ul`
- `.keywords`（`.k`）… 章末のキーワードタグ
- `.pager` … 前後ページ送り
- index 専用：`.bibliography`（書誌情報）/ `.book-overview` / `.part-grid`＋`.part-card` / `.section-label`
- index 専用：`.persona-grid`＋`.persona`（ターゲット読者カード。`.emoji`＋`h3`＋`p`）
- index 専用：`.ai-analysis`（**AIによる分析の囲み**。`.ai-badge` で本書外を明示、`.path-row`＝`.who`＋`.route` で立場別の読み順を示す）

## 強調

- `<mark>` … 文中の最重要句のハイライト（章につき数か所まで。乱用しない）
- `<span class="term">…</span>` … 重要な専門用語（点線下線＋アクセント色）

## コールアウト

- `.in-a-nutshell`（`.emoji`＋`.label`「この章を一言で」＋`p`）… 章冒頭の要旨
- `.callout.why` … 「なぜそうなるか」の補足説明
- `.callout.tip` … 言い換え・読み解きの補助
- `.callout.warn` … 注意・落とし穴
- `.callout.supplement`（**本書外マーク**）… 本にない補足。`.ctitle` に「補足（本書外）」と書く。本書由来でない情報はこれで囲む

インラインで本書外の語を示す場合は `<span class="supp">語<span class="supp-tag">補足</span></span>`。

## 図解

- `.versus`（`.side.a` / `.vs` / `.side.b`）… A対Bの対比（Dev対Ops、Before対After など）
- `.steps`（`.step`＋`.badge`、悪化の連鎖は `.steps.escalate`）… 段階・手順・幕
- `.metrics`（`.metric`＋`.big`/`.name`/`.dir`、`.up`/`.down`）… 指標カード
- `.flow`（`.node`＋`.arrow`）… 横方向のフロー図
- `.ways-map`（`.way-row`、`--wc` で色指定）… 並列する原則・分類のまとめ図

## カスタム図解（SVG）― 本の情報を図にする

上記の標準部品で表せない構造（独自の記法・図形・関係図など）を、本書を分かりやすくする目的でのみ、インライン SVG で自作してよい。**新しい情報を足す手段ではない。**

**作ってよい条件（すべて満たすこと）**

- 本書自身が**図・記法・要素間の関係**として示している内容である（例：分類を形で表す記法、要素どうしの関係図）。
- 既存部品（`.versus` / `.flow` / `.ways-map` / `.steps` 等）では十分に表せない。

**抑制（過度な挿入の禁止）**

- まず既存部品で表せないか検討する。表せるならカスタム図は作らない。
- 点数は**ページの長さではなく「図にすべき構造の数」で決める**。本書が図・記法・関係として示す構造1つにつき図は1点までとし、同じ構造を複数の図で繰り返さない（情報量が多いページでも、構造が増えない限り図は増やさない）。
- **必要性テスト**を満たす図だけ作る：「この図がなければ、本文＋標準部品では同じ理解に届かない」と言えること。装飾・箇条書きの焼き直し・要点の水増しは作らない。

**守る規則（`writing-rules.md` に準拠）**

- 図に含めてよいのは**本書にある情報だけ**。本にない関係・数値・分類・主張を図にしない。
- 本書由来でない視覚要素（**識別用の配色・レイアウト上の配置**）は、`figcaption` に「（配色は識別用であり原書由来ではない）」のように明示する。形・記法そのものが本書由来なら、その旨の注記は不要。
- 図は本文の代替ではなく補助。記号・線種を使うときは短い凡例と `figcaption` を添え、図単体でも誤読されないようにする。
- 実装はインライン SVG（`viewBox` で可変、画像ファイル不要）。色・線は既存トークン（`var(--line)` / `var(--card)` / `var(--ink-faint)` 等）に追従させ、ページのアクセント色と調和させる。

## 詳細モードの構成パターン（mode=詳細のとき）

詳細モードでも**新しい見た目部品は追加しない。** 既存部品の組み合わせで各論点を縦に厚くする。

- **トピックの推奨並び**：`h3.ic`（要点見出し）→ `.summary`（要旨）→ **論拠展開段落（通常の `<p>`）** → `ul`（箇条書き）→ 必要なら `.callout.why`（「なぜそう言えるか」の核）。見出し・トピックの骨格は標準モードと同じに保つ。
- **論拠展開段落**は通常の段落（`<p>`）で書く。内容は `writing-rules.md` の「7. 詳細モードの論拠展開」に従い、著者の論拠・前提・筋道に限る。
- **「なぜそう言えるか」**を1文で締めたいときだけ `.callout.why` を使う（既存部品）。トピックごとに必須ではない。
- **専門用語の短い定義**は `.callout.supplement`（本書外マーク）またはインラインの `<span class="supp">語<span class="supp-tag">補足</span></span>` を使う。背景・論評の拡張には使わない。
- **カスタム図のルールは不変**：深掘りを理由に図を増やさない（「本書が図・記法・関係で示す構造に限り1構造1点」を維持）。

## 使い分けの目安

| 見せたいもの | 部品 |
|---|---|
| 章の要旨を最初に | `.in-a-nutshell` |
| 要点と解説 | `.topic` |
| 2つの立場・状態の対比 | `.versus` |
| 段階・連鎖・手順 | `.steps`（悪化は `.escalate`） |
| 指標の方向（増減） | `.metrics` |
| 一連の流れ | `.flow` |
| 並列する複数の原則 | `.ways-map` |
| 本書独自の記法・関係図（既存部品で表せないもの） | カスタム SVG（上記条件・抑制ルールを守る） |
| 本にない補足 | `.callout.supplement` / `.supp` |

絵文字は各部品の `.emoji` 枠やアイコン位置にのみ置く。解説文に混ぜない。
